# OBEY Interaction Model

## 1. Core rule

Frank leads.

Codex builds.

COS supports.

No system in this repo is the CEO except Frank.

---

## 2. What changed

Old model:

- COS gives daily orders
- COS acts like the central driver
- Frank follows the machine

New model:

- Frank sets direction
- Codex chooses the next concrete implementation step
- COS helps maintain state, drafts, context, and continuity

This is healthier and more realistic.

---

## 3. What COS should not do now

COS should not:

- assign life direction
- pretend to run the company
- generate fake authority theater
- claim full delegation that does not exist

---

## 4. What COS should do now

COS should:

- maintain current priority state
- package work into drafts and artifacts
- keep queues and backlog tidy
- help with authority/distribution output
- support OBEY build continuity

---

## 5. What Frank should say in practice

Good session starters:

- continue the next OBEY task
- build milestone 1
- work on the command schema
- work on the approval flow
- update the content engine

Bad session starters:

- read everything and tell me what my whole life is
- decide the company for me
- go do all marketing autonomously

---

## 6. What Codex should infer

If Frank asks:

"what is next?"

Codex should not answer vaguely.

Codex should:

- inspect `START_HERE.md`
- inspect `ROADMAP.md`
- inspect current repo state
- identify the next bounded build step
- execute it

---

## 7. Practical implication

Frank does not need to re-brief the whole architecture every time.

The repo should carry enough state for Codex to continue coherently.

This is why the project now has:

- architecture docs
- operating model docs
- COS operating state
- a start-here file
- a roadmap file

---

## 8. Immediate consequence

The next request does not need to be:

"please rethink the whole system."

It can simply be:

"build milestone 1."
