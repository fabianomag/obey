# COS Model

**Status:** draft v0  
**Date:** 2026-04-19  
**Purpose:** define what COS actually is, how it helps, and why it is not "just text" or "the boss."

---

## 1. Core definition

COS is not the product.

COS is not the founder's boss.

COS is not a generic skill marketplace.

COS is an internal proactive operating layer that helps Frank use frontier models with more continuity, initiative, and leverage.

In simple terms:

- OBEY is what the market sees
- SGOPS is what documents and distributes the thinking
- COS is the internal layer that helps the machine keep moving

---

## 2. Why COS exists

Raw LLMs are powerful but mostly reactive.

They respond well when the human already knows:

- what to ask
- what context matters
- what output format matters
- what the next step should be

That means the real missing layer is often not intelligence.

It is initiative under constraints.

COS exists to reduce that gap.

---

## 3. What COS is not responsible for

COS should not be judged by impossible criteria.

It is not responsible for:

- inventing the company strategy alone
- replacing founder judgment
- fully autonomous internet operation
- magically solving every business problem with one prompt
- becoming a complete agency without setup, assets, or rules

If judged by those standards, it will always look like failure.

---

## 4. What COS is responsible for

COS should help by being proactively useful inside bounded operating loops.

Its job is to:

- notice what the current priority is
- maintain working context
- propose the next best actions
- package work into usable artifacts
- keep queues, briefs, and drafts moving
- reduce repeated human orchestration

This is different from being a chatbot.

This is closer to an internal operating system.

---

## 5. The key distinction: proactive, not dominant

COS should be more proactive, but not authoritarian.

That means:

- it can suggest
- it can prepare
- it can remind
- it can assemble
- it can queue
- it can draft

But:

- Frank still decides direction
- Codex still implements systems
- OBEY still carries the public wedge

COS is a force multiplier, not the source of sovereignty.

---

## 6. Current highest-value mode

The current biggest pain is not "website generation."

The current biggest pain is:

**lack of a reliable internal marketing/distribution engine.**

So the first mode COS should become great at is:

## Marketing Operations Mode

This does not mean "COS is hardcoded to be an agency forever."

It means that, for the current phase, the most valuable operating mode is distribution.

---

## 7. COS operating modes

COS should support multiple modes over time.

### Mode A - Marketing Operations

Purpose:

- turn real work into public proof
- keep authority surfaces alive
- reduce distribution paralysis

Examples:

- convert work logs into post drafts
- maintain a weekly content queue
- link SGOPS, OBEY, and GitHub outputs
- extract proof snippets from real projects
- propose CTAs and publishing priorities

### Mode B - Product Operations

Purpose:

- support OBEY build execution
- keep specs, decisions, and runtime context coherent

Examples:

- summarize product decisions
- maintain project briefs
- prepare comparison tests
- propose next implementation steps

### Mode C - Research Operations

Purpose:

- support SGOPS as a lab surface
- keep track of insights, failures, and comparisons

Examples:

- turn experiments into notes
- cluster themes
- generate draft essays
- map recurring pain patterns

### Mode D - Client Operations

Purpose:

- support real-world service delivery when needed

Examples:

- prepare briefs
- capture approvals
- maintain history
- package deliverables

Important:

COS is not all modes at once every day.

One mode should dominate based on the current weekly objective.

---

## 8. Current recommendation

For the current phase:

- primary COS mode = Marketing Operations
- secondary COS mode = Product Operations
- SGOPS remains the lab and publishing surface
- OBEY remains the public wedge

This matches Frank's actual pain instead of forcing fake generality.

---

## 9. How COS can be real if it is "natural language"

COS should not be understood as only prose.

It should be understood as a hybrid system with five layers.

### Layer 1 - Natural-language policy

This is where founder intent, style, constraints, and operating rules live.

Examples:

- what counts as proof
- what not to publish
- how to position OBEY
- how to write for SGOPS

### Layer 2 - Structured state

This is where repeatability comes from.

Examples:

- project briefs
- decision logs
- content queue
- campaign map
- weekly priorities
- approval records

These can live in Markdown, YAML, JSON, or other simple files.

### Layer 3 - Templates

This is where outputs become consistent.

Examples:

- post template
- case-study template
- weekly content pack template
- wedge proof template

### Layer 4 - Connectors and scripts

This is where the system touches the outside world.

Examples:

- publish adapter
- webhook handler
- asset renderer
- CLI commands

### Layer 5 - Triggers and cadence

This is where proactivity comes from.

Examples:

- daily review
- weekly packaging
- event-based reminders
- new artifact detection

So no, COS is not "just text."

It is a natural-language-controlled operating layer sitting on top of structured state and executable connectors.

---

## 10. Where proactivity actually comes from

Proactivity should not mean "the model wakes up and does random things."

Real proactivity comes from:

- current priority
- queue state
- explicit triggers
- missing artifacts
- stale work detection
- approval status

Examples:

- "There is no SGOPS note for this week's work."
- "Three site updates have no proof post yet."
- "The content queue has only one approved item left."
- "The OBEY repo changed but no public update was drafted."

This is much more realistic than vague autonomy.

---

## 11. The correct mental model

The wrong mental model is:

"COS is a magical autonomous employee."

The better mental model is:

"COS is a proactive orchestration layer that keeps bounded loops moving."

That is enough to create real leverage.

---

## 12. What COS should do this month

If we were being strict, COS should only be asked to help with a few high-value loops:

- maintain an authority backlog
- turn work into weekly content packs
- extract proof from product progress
- connect SGOPS, OBEY, and GitHub artifacts
- keep a running map of current positioning and approved language

That alone would already make it feel much more alive and useful.

---

## 13. Strategic conclusion

Yes, COS can become coherent enough for this vision.

But only if it stops trying to be everything and becomes:

- internal
- mode-based
- proactive through triggers
- grounded in structured state
- connected to real execution surfaces

That is how COS stops feeling like a vague prompt and starts feeling like infrastructure.
