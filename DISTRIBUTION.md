# OBEY Distribution Architecture

## 1. The real problem

Frank does not only need a product.

Frank also needs a repeatable way to turn real work into public proof.

Right now the bottleneck is not "lack of ideas."

The bottleneck is:

- no reliable publishing engine
- no stable content workflow
- too much expectation of agent autonomy on hard external platforms

That means the immediate problem is not "make COS run all marketing alone."

The immediate problem is:

**build a content and proof pipeline that works even before full autonomy exists.**

---

## 2. Hard truth

COS has not yet proven that it can:

- create accounts reliably
- configure platform APIs reliably
- operate Instagram/Facebook end-to-end without supervision

That is not a moral failure.
That is a product boundary.

Those tasks are among the hardest classes of agent work because they involve:

- authentication friction
- anti-bot systems
- changing interfaces
- platform restrictions
- brittle external dependencies

So this should not be the first public proof of value.

---

## 3. What authority should mean in phase one

Authority in phase one should not mean:

"my AI runs the whole internet by itself."

Authority in phase one should mean:

"I have a system that consistently turns business context into useful output faster and more coherently than people do with raw AI alone."

That is already valuable and much easier to prove.

---

## 4. The obedience ladder

Not all obedience is the same.

### Level 1 - Draft obedience

Given a source, the agent produces the requested output in the requested format.

Examples:

- post draft
- article outline
- CTA options
- content summary

### Level 2 - Context obedience

The agent keeps the right references, decisions, tone, and positioning across outputs.

Examples:

- uses the right offer
- respects approved claims
- reuses project references
- stays aligned with the brand angle

### Level 3 - Workflow obedience

The agent updates files, project state, and linked surfaces in a controlled workflow.

Examples:

- creates a new article file
- updates a content queue
- writes social captions to a publish folder
- links SGOPS and OBEY assets together

### Level 4 - Platform obedience

The agent publishes through external systems with approval checkpoints.

Examples:

- posts to a CMS
- schedules content
- uploads prepared assets

### Level 5 - Full external autonomy

The agent creates accounts, configures APIs, and runs third-party surfaces with little supervision.

This is the hardest level and should not be the initial promise.

Important:

Frank does not need Level 5 to have a real business.
Frank needs Levels 1 to 3 working consistently first.

---

## 5. Distribution system for phase one

The system should look like this:

```text
Real work
  -> capture
  -> content pack
  -> approval
  -> publish
  -> cross-link
  -> archive proof
```

### Input: real work

Sources:

- client conversations
- site changes
- before/after results
- decisions made during builds
- failures and fixes
- experiments with Claude Code and Codex

### COS responsibility

COS should turn raw material into a structured content pack.

That pack can include:

- one short post
- one long post or article draft
- one proof snippet
- one CTA
- one cross-link suggestion

### Frank responsibility

Frank remains responsible for:

- truth
- selection
- approval
- final taste
- deciding what is worth publishing

### Publisher responsibility

A thin publishing layer can remain manual at first.

That is acceptable.

Manual publishing is not failure if the expensive thinking and packaging work is already compressed by COS.

---

## 6. Owned-media stack

The initial stack should be intentionally small.

### Primary surfaces

- SGOPS site for essays, learnings, and failures
- OBEY site for product framing and OSS entrypoint
- GitHub for open-source proof

### Secondary surface

Choose one social channel to keep alive consistently.

Candidates:

- LinkedIn for authority and business context
- X for agent-builder discussion
- Instagram for brand presence and visual packaging

Important:

If energy is limited, do not try to fully operate all three at once.
One healthy channel is better than three dead channels.

---

## 7. What COS should actually do now

COS should not be asked to "do marketing."

That is too vague and too large.

COS should be asked to perform bounded jobs:

- convert a transcript into 3 post angles
- convert a site update into a before/after proof post
- produce a weekly content pack from work completed
- maintain a queue of approved draft ideas
- keep links between SGOPS, OBEY, and GitHub artifacts

This is tractable.
This is measurable.
This is much closer to real obedience.

---

## 8. Minimum viable content engine

The first useful content engine should produce, every week:

- 1 SGOPS note
- 1 OBEY proof update
- 3 short-form posts derived from real work

That is enough to create visible motion.

Anything beyond that is optional until the system is stable.

---

## 9. What not to depend on yet

Do not make the whole strategy depend on:

- automatic Instagram account creation
- automatic Facebook configuration
- end-to-end hands-free social posting
- broad multi-platform orchestration

Those can come later.

They are not the wedge.
They are optional amplifiers.

---

## 10. Strategic conclusion

Frank should not measure competence by whether COS can operate Facebook on its own.

That is the wrong benchmark.

The correct benchmark is:

- does COS help convert real work into public proof consistently?
- does OBEY clarify why that system behaves better than raw agent usage?
- does SGOPS accumulate evidence and language around the method?

If yes, the system is moving.

If no, the next task is not "more ideas."
The next task is to tighten the workflow until weekly proof starts appearing.
